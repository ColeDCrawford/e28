Comprehensive build

For our second build approach, we won’t start with an existing project; instead we’ll build a new application from scratch with the intention of making it fully Vue-powered with complete SPA (Single Page Application) support.

To accomplish this, we’ll still use NPM, Webpack, and Babel.

However, instead of using Laravel Mix, we’ll use Vue CLI, a Vue command line tool that will a) set up our project directory and b) configure Webpack.

Vue CLI

Vue Cli is a command line program you can use to make Vue projects.

Install it globally on your computer with the following NPM command:

$ npm install -g @vue/cli
FYI

Mac users - if you encounter a EACCES: permission denied error when running the above command, try adjusting the permissions on your computer’s global node_modules directory: 

$ sudo chown -R $USER /usr/local/lib/node_modules

Once Vue CLI is installed, you should have access to the command vue from any directory on your computer.

Navigate into your e28 directory and use vue create to create a new project called week9:

$ vue create week9

In the prompt that follows, hit Enter to select the first option, Default ([Vue 2] babel, eslint) 

Vue CLI v4.5.8
? Please pick a preset: (Use arrow keys)
❯ Default ([Vue 2] babel, eslint) 
  Default (Vue 3 Preview) ([Vue 3] babel, eslint) 
  Manually select features

Vue CLI should then run through a series of steps to set up your project, resulting in a directory called week9 with the following contents: 

.gitignore
README.md
babel.config.js
node_modules/
package-lock.json
package.json
public/
src/

Navigate into your new week9 directory and run the vue-cli-service serve command which is aliased via serve in package.json:

$ npm run serve

This will start a webpack dev server accessible via http://localhost:8080/ that shows this default start page:

Like with our integrated approach, using this webpack dev server replaces our XAMPP/Apache server and gives us the advantages of hot module reloading.

Understanding how the dev server works

If you View Source in the browser when running your app through the dev server, you’ll see it pulls in 2 JavaScript files:

<link href="/js/app.js" rel="preload" as="script">
<link href="/js/chunk-vendors.js" rel="preload" as="script">

Given this, you’d expect to see files in a directory at /e28/week9/js; however, no such directory exists.

This is because the dev server is not actually compiling/writing these files to disk— instead it’s serving them from the webpack dev server’s dynamic memory (e.g. RAM). (If you visit http://localhost:8080/webpack-dev-server you can see all the assets being served from dynamic memory).

Building for production

The system of serving up our assets from the webpack dev server’s memory won’t work on production. Instead, we need our build system to write our compiled files to disk so they can be synchronized to production via git.

To do that, run the vue-cli-service build command which is aliased via build in package.json:

$ npm run build

This command will produce a production-ready bundle in a /dist directory, with minification for JS/CSS/HTML and auto vendor chunk splitting for better caching.

By default, Vue CLI apps are set to ignore this directory (via git’s .gitignore file), as one approach to managing your compiled files is to build them directly on production.

Because we’re not using the “build on production” approach, we need to remove the line /dist from .gitignore so that we can track it via git.

After you do that, add/commit/push all your changes to Github so we can deploy this application to production.

Deploying on production

On production, navigate into your /e28 directory and run git pull to get the latest changes.

To test your project out, configure a subdomain in your server’s VirtualHosts configs (/etc/apache2/sites-enabled/000-default.conf):

<VirtualHost *:80>
    ServerName e28week9.yourdomain.com
    DocumentRoot "/var/www/html/e28/week9/dist"
    <Directory "/var/www/html/e28/week9/dist">
        AllowOverride All
        Options -Indexes
        Require all granted
    </Directory>
</VirtualHost>

Note how we’re not just pointing to week9/; instead we’re pointing to week9/dist where the compiled files were written.

After restarting Apache (service apache2 restart) you should be able to access your example project at http://e28week9.yourdomain.com.

Understanding how our application is structured/configured

When we build an application using Vue CLI there are many pre-configured options that work together to make our application and build system function.

For insight into these pre-configured options, run vue inspect within your project to see the webpack config that Vue CLI generates.

Skimming through these settings you can find configs that set...

  • /src/main.js as the entry point to our application
  • /dist as the directory to which our built files are written
  • /public/index.html as the template file webpack uses to generate our index file
  • etc.

To learn more about each of these settings, as well as learn how to customize them if needed, check out this Vue CLI documentation: https://cli.vuejs.org/config/#vue-config-js
