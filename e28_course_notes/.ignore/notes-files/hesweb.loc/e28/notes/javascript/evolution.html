JavaScript Evolution

Early role of JavaScript: Mostly for effect

  • Animated progress bars
  • Image slide slows
  • Hover effects
  • Tabbed interfaces
  • Collapsible content
  • Dropdown menus
  • Etc.

Enter: Ajax

Asynchronous JavaScript XML

Bundle of technologies:

  • HTML/CSS
  • The XMLHttpRequest API for asynchronous communication with the server
  • JSON, XML, or HTML to format the data transmitted to/from the server
  • The DOM API to alter the page as a result of the server communication
  • JavaScript to bring everything together

Early on, Ajax/JavaScript was used to enhance one or two features of a page.

Example: Autocomplete search boxes

But sites grew into robust applications, heavy with Ajax/JavaScript.

To facilitate the increasingly complex tasks of these sites, JavaScript libraries like jQuery became very popular.

Vanilla JS to jQuery back to Vanilla JS

  • 2006
  • “Solution” to vanilla JS
    • Abstracted/simplified:
      • HTML Dom tree traversal and manipulation
      • CSS manipulation
      • Event handling
      • Ajax
      • JSON parsing
      • And more...
    • Provided cross-browser compatibility

jQuery is dead, except not really

  • Vanilla JS and browser’s JS engines have evolved, rendering jQuery unnecessary in new projects
  • A large percentage of web sites still in use today were built with jQuery
  • JS frameworks have taken over for many of the tasks jQuery used to facilitate, providing us an improved approach to complex application development

JavaScript frameworks

As websites evolved into increasingly complex, dynamic interfaces, JavaScript frameworks started showing up that addressed many of the common functional and organizational needs of a single-page application.

FYI

Libraries vs. Frameworks:

  • Libraries like jQuery provide functionality
  • Frameworks like Vue, React, Angular provide functionality and define the application design

JS framework key features and designs

  1. Data-driven
  2. Components

1. Data-driven

Previously, JS interfaces were built in a very imperative fashion, where working with the DOM was done very explicitly.

Before - explicit/imperative style

<input id='message'>

// Explicitly target the div in the DOM and change its contents
document.getElementById('message').value = 'Hello world!';

New - implicit/declarative style

<input id='message' bind='message'>

// Because the message variable is bound to the div, 
// we just have to update the message and it will automatically update the contents of the div
message = 'Hello world';

(The above code is simplified and not specific to any one JS framework)

2. Components

  • Like DIY HTML elements

  • Standard HTML elements, e.g. <h1>, <ul>, <a> etc.

  • Components = DIY HTML elements, e.g. <slideshow>, <user-profile>, <shipping-calculator>

Example: a component called shipping-calculator could be defined with the following template:

<div class='calculator'>    
    <input type='text' name='zipCode' v-bind='zipCode'>

    <ul>
        <li v-for='options in shippingOptions'>{{ option.carrier }} ${{ option.cost }}</li>
    </ul>
</div>

Then used in your HTML like so:

<shipping-calculator>

Benefits of components:

  • Focus on each part of your interface in isolation
  • Encourage reusability through modularity
  • Discover a large ecosystem of components shared amongst developers

Vue.js

  • Created by Evan You, former Google developer
  • Open-source, crowd funded
  • Goal: extract the things he (Evan) liked about frameworks like Angular and React but exclude the things he didn’t like and felt weighed other frameworks down (ref).

Release timeline:

  • Angular October 2010
  • React May 2013
  • Vue February 2014

Vue.js benefits/highlights:

  • Lean, with a minimal core
  • Progressive - can be incorporated into existing projects, or part of a project; not a “monolithic” framework
  • Incrementally adoptable - supporting libraries provide additional functionality (e.g. routing, state management)
  • Beginner friendly
