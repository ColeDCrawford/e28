Component Props

An option you can add to components are props, which provide a way to pass data to child components.

The following code defines two props, number and winner: 

// Component Vue instance
Vue.component('round-detail', {
    // [...]
    // Define 2 props
    props: ['number', 'winner'],
    // [...]
});

You can use the props in the component template just like you would a regular data property. In this example we use text interpolation ({{ }}) to output the prop values:

<script type='text/x-template' id='round-detail'>
    <div v-if='!deleted'>
        <ul>
            <li>Round #: {{ number }}</li>
            <li>Winner: {{ winner }}</li>
        </ul>
        <button @click='deleteRound'>Delete round</button>
    </div>
</script>

Then, we set the prop values via custom HTML attributes on the component. To start, we’ll hard-code some example values:

<div id='app' v-cloak>
    <round-detail number=1 winner='Player'></round-detail>
    <round-detail number=2 winner='Player'></round-detail>
    <round-detail number=3 winner='Computer'></round-detail>
</div>

Prop case conventions

If your prop names require multiple words, they can be defined in your JavaScript using camelCase casing, but they should be written in the HTML as kebab-case.

For example, let’s imagine our prop winner (1 word) was instead named who-won (2 words).

When defining the prop, we could reference it as whoWon (camelCase):

// Component Vue instance
Vue.component('round-detail', {
    // [...]
    // Define 2 props
    props: ['number', 'whoWon'],
    // [...]
});

But when using it in the HTML, we have to write it as who-won (kebab-case):

<round-detail number=1 who-won='Player'></round-detail>

This is necessary because HTML attributes are always translated to lowercase, so if we were to use the camelCase style as it was defined, whoWon, it would actually render as whowon in the HTML. This would produce the following warning in the console:

[Vue tip]: Prop "whowon" is passed to component <Anonymous>, but the declared prop name is "whoWon". 
Note that HTML attributes are case-insensitive and camelCased props need to use their kebab-case equivalents when using in-DOM templates. 
You should probably use "who-won" instead of "whoWon".

Vue uses automatic translation on props, so when you define it as whoWon, it can be used as who-won.

Dynamic props

When the values for your props are static, like in the above examples, you can define them using regular HTML custom attributes.

<round-detail number=1 winner='Player'></round-detail>

If the values are dynamic, however, you should use v-bind.

For example, imagine we had a data prop in our root Vue instance that contained the details about all the rounds:

// root Vue instance
let app = new Vue({
    el: '#app',
    data: {
        rounds: [
            { number: 1, winner: 'Player' },
            { number: 2, winner: 'Player' },
            { number: 3, winner: 'Computer' },
        ]
    }
});

When we iterate through this data, invoking a round-detail component for each piece, we could pass in the appropriate props (number and winner) using v-bind: 

<div id='app' v-cloak>
    <round-detail 
        v-for='round in rounds' 
        v-bind:number='round.number' 
        v-bind:winner='round.winner'>
    </round-detail>
</div>

Keys

The above code technically works, but it will produce a warning:

[Vue tip]: <round-detail v-for="round in rounds">: component lists rendered 
with v-for should have explicit keys. 
See https://vuejs.org/guide/list.html#key for more info.

As the warning notes, it’s expected that when you render lists of components, each component should have an explicit key, which is some unique value to represent that component. These keys are used to help Vue most efficiently re-order data (should the data need to be re-ordered).

For this example, knowing the round numbers should be unique, we can use that as our key to make the warning go away:

<div id='app' v-cloak>
        <round-detail 
            v-for='round in rounds' 
            v-bind:key='round.number' 
            v-bind:number='round.number' 
            v-bind:winner='round.winner'>
        </round-detail>
</div>

For a more detailed explanation of why the key is necessary, here’s a paragraph from the Vue API on keys:

The key special attribute is primarily used as a hint for Vue’s virtual DOM algorithm to identify VNodes when diffing the new list of nodes against the old list. Without keys, Vue uses an algorithm that minimizes element movement and tries to patch/reuse elements of the same type in-place as much as possible. With keys, it will reorder elements based on the order change of keys, and elements with keys that are no longer present will always be removed/destroyed.

Prop types and default values

So far our props have been simple strings:

props: ['number', 'winner'],

We can expand on this using objects that allow us to specify the expected data type for each prop and/or a default value:

props: {
    // `number` has an expected data type of Number, and it defaults to 0
    number: {
        type: Number,
        default: 0
    },
    // `winner` has an expected data type of String, with no default set
    winner: String,
},

If a prop value contradicts it’s expected data type, Vue will show a warning in the console.

When specifying data types, the following options exist:

  • null - Will pass any type validation; equivalent to not setting a type
  • String
  • Number
  • Boolean
  • Array
  • Object
  • Date
  • Function
  • Symbol
