Components

  • Review: JS framework key features and designs : Components
  • “DIY HTML elements”
  • Reusable Vue instances with their own data, methods, and computed properties.

Initializing components

FYI

The following procedures are specific to Vue v2; for v3 details on component initialization, read Vue 3: Components

One way components can be created is via the Vue.component method:

Vue.component(id, options)

For example, let’s create a round-detail component that will be used to display the various results from rounds of a hypothetical game:

JavaScript:

// Component Vue instance
Vue.component('round-detail', {
    // Note how the data property receives a function in a Component
    data() {
        return {
            deleted: false,
        }
    },
    template: `
    <div v-if='!deleted'>
        <ul>
            <li>Round #: TBD</li>
            <li>Winner: TBD</li>
        </ul>
        <button @click='deleteRound'>Delete round</button>
    </div>
    `,
    methods: {
        deleteRound() {
            this.deleted = true
        }
    }
});

// root Vue instance
let app = new Vue({
    el: '#app'
});

HTML body:

<div id='app' v-cloak>
    <round-detail></round-detail>
</div>

The following sections break down the details of what’s happening in the above code.

Options

A component is just another Vue instance, so it shares a lot of similarities with the root Vue instance we’ve been working with so far.

For example, you’ll see many of the same options: data, methods, watch, computed, and lifecycle hooks.

One option you won’t see is el, which we used with our root Vue instance to define the HTML element that would be our template. With components, we’ll instead define our template using the template option. More on that in a bit.

Finally, note that the data option in a component must be a function that returns our option of data - this ensures each instance of the component will have its own independent data.

Naming

Component names should be multi-word and hyphen-separated (e.g. round-detail) so that they follow the rules of HTML custom element names:

“[custom element names should] contain a hyphen, used for namespacing and to ensure forward compatibility (since no elements will be added to HTML, SVG, or MathML with hyphen-containing local names in the future).” -w3c spec #valid-custom-element-name

We could have named the component in our JavaScript using PascalCase (RoundDetail) but it should be written as <round-detail></round-detail> in DOM templates. Vue uses automatic translation to resolve the difference.

Ref: Vue.js style guide: Multi-word component names

Organization

The approach shown in the above example created a globally registered component; as long as it’s declared before the root Vue instance, it will be globally available for use within that instance.

Vue instances can be nested inside of one another. In our example, the round-detail component was nested in the root Vue instance. Eventually we’ll nest components inside of other components.

To describe these nesting patterns, we use terms like “parent” and “child” when describing Vue instances and their relationship to one another.

In our example so far we can consider the root Vue instance to be the parent, and round-detail a child component of that instance since it’s nested within it.

Observe how components show up in Vue dev tools as children of the root Vue instance, and they each have their own data and property values.

Templates

In Vue, templates are how you define the HTML markup your Vue instances will use.

So far, with our root Vue instances we’ve been using a DOM template strategy: We define our template in our HTML file, give it an id, and then reference it in the Vue instance via the el option.

DOM template strategy with root Vue instance:

<!-- The body of our HTML file -->
<div id='app' v-cloak>
    <round-detail></round-detail>
</div>

// root Vue instance
let app = new Vue({
    el: '#app'
});

In our first component above, however, we take a different approach by defining the template via the template option in the component instantiation. 

// Component Vue instance
Vue.component('RoundDetail', {
    //[...]
    template: `
        <div v-if='!deleted'>
            <ul>
                <li>Round #: TBD</li>
                <li>Winner: TBD</li>
            </ul>
            <button @click='deleteRound'>Delete round</button>
        </div>
        `,
    // [...]
}

Specifically, we defined it using a template literal using backticks (``) for legibility.

The template option can also accept regular string values ('' or ""), but you can’t add line breaks in regular strings, so the HTML markup can be challenging to read/write.

As an alternative to defining your template as a template literal or a string, you can define your template in your HTML file by embedding it in a script element with type text/x-template: 

<!-- TEMPLATE FOR OUR COMPONENT -->
<script type='text/x-template' id='round-detail'>
    <div v-if='!deleted'>
        <ul>
            <li>Round #: TBD</li>
            <li>Winner: TBD</li>
        </ul>
        <button @click='deleteRound'>Delete round</button>
    </div>
</script>

<!-- TEMPLATE FOR OUR ROOT VUE INSTANCE -->
<div id='app' v-cloak>
    <round-detail></round-detail>
</div>

Then pass the id of this element to your template option

Vue.component('round-detail', {
    // [...]
    template: '#round-detail',
    // [...]}
});

All of the above approaches have limitations:

  • Template literals are an ES6 feature, which may not be supported by older browsers.
  • Template strings are messy to work with because you can’t use line breaks.
  • Both template literals and strings lack proper HTML syntax highlighting, as they’re just strings embedded in JS.
  • Defining the template in the HTML via an x-template ties it to that HTML file, making it impossible to reuse that component in other files. As we’ll soon see, a key benefit of components is reusability so this obviously isn’t ideal.

Eventually, we’ll look at something called single file components and a build process to transpile modern ES6 code into more ubiquitously supported ES5 code. Both of these things will address all of the above issues. For now, we’ll proceed forward knowing the limitations and understanding we’ll address them in the future.

Single parent element rule

Note that component templates must have a single parent element.

In our example, the div is the single parent element:

<script type='text/x-template' id='round-detail'>
    <div v-if='!deleted'>
        <ul>
            <li>Round #: TBD</li>
            <li>Winner: TBD</li>
        </ul>
        <button @click='deleteRound'>Delete round</button>
    </div>
</script>

If we were to remove it...

<script type='text/x-template' id='round-detail'>
    <ul>    
        <li>Round #: TBD</li>
        <li>Winner: TBD</li>
    </ul>
    <button @click='deleteRound'>Delete round</button>
</script>

...we’d get a Vue warning in the console because we now have two independent elements (ul and button):

vue.js:634 [Vue warn]: Error compiling template:

Component template should contain exactly one root element. 
If you are using v-if on multiple elements, use v-else-if to chain them instead.

5  |  <li>Winner: TBD</li>
6  |  </ul>
7  |  <button @click='deleteRound'>Delete round</button>
FYI

Note: The single parent element rule no longer applies in Vue v3.

Reusability

Components allow us to break up aspects of our program into smaller, modular parts. Because of this, one of the advantages of components is reusability:

<div id='app' v-cloak>
    <round-detail></round-detail>
    <round-detail></round-detail>
    <round-detail></round-detail>
    <round-detail></round-detail>
</div>

Of course, our design so far has 2 major limitations:

  1. Because our template for the round-detail component is defined in the HTML file, it’s tied to that file and not re-usable across other files our application might have. We’ll address this later by truly modularizing the component code into something called single-file components.
  2. The Round and Winner for each results is hard-coded as TBD when it should be unique for each result. We’ll address this in the next note set with something called props.
