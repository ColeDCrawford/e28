Building ZipFoods

To demonstrate building an application with Vue.js, we’re going to work on a hypothetical project called ZipFoods:

“ZipFoods is your one-stop-shop for convenient online grocery shopping in the greater Boston area.”

Because this app will be built from the ground up, we’ll use the “comprehensive” approach and initialize the project with Vue CLI.

To begin, we’ll warm-up by setting up some features to review concepts we’ve already covered. Then, we’ll begin to improve on these features and pull in more advanced concepts.

Setup

$ vue create zipfoods

Once the application is created:

  • Skim through the file structure, noting the following key files/locations:
    • /public/index.html Main HTML file that contains the #app div to which the root component will be mounted
    • /src/main.js Main JS file that will mount the root component, App.vue
    • /src/components Where component files will go
    • /src/components/App.vue Root component
  • Update the <title> element in /public/index.html to read ZipFoods
  • Download zipfoods-assets.zip. From this download...
    • Put images in /src/images/
    • Put favicon.ico in /public/
    • Put styles/zipfoods.scss in /src/assets/scss/
  • Update the root component (App.vue):
    • Remove the HelloWorld component
    • Clear the default styles
    • Update the logo to display the ZipFoods logo (@/assets/images/zipfoods-logo.png) with an id of logo.
    • Below the logo, display this paragraph of text: ZipFoods is your one-stop-shop for convenient online grocery shopping in the greater Boston area.
  • Run npm install -D sass-loader sass to install the webpack sass-loader (ref)
  • In App.vue import the provided zipfoods.scss file:
<style lang='scss'>
@import '@/assets/scss/zipfoods.scss';
</style>

Once you’ve completed the above steps, your application should look like this:

FYI

In the work done above, we made reference to two paths:

  1. The logo at @/assets/images/zipfoods-logo.png
  2. The .scss file at '@/assets/scss/zipfoods.scss'

In both these examples, we referenced the path using an @ sign, which is an alias Vue CLI configures with webpack that points to your src/ directory.

ShowProduct component

Our first goal with this application is to display a list of products on the home page.

To accomplish this, first create a ShowProduct (singular) single file component that will:

  • accept a prop called product
  • display an individual product
FYI

Revisit the Single File Component note set for a refresher on the structure of your SFCs.

Here’s a template you can use:

<!-- src/components/ShowProduct.vue -->
<template>
    <div class="product">
        <div class="product-name">{{ product.name }}</div>
        <img
            class="product-thumb"
            :src="require('@/assets/images/products/1.jpg')"
        />
        <p class="product-description">{{ product.description }}</p>
        <div class="product-price">${{ product.price }}</div>
    </div>
</template>

Note that the image src is set using Vue loader’s require method, which forces the asset to be loaded via a webpack module request, ensuring that our images will end up in our built dist/ folder.

Ref: Vue Loader: Asset URL Handling

ShowProducts component

Next, create a ShowProducts (plural) component that will:

  • display a list of products using the above-created ShowProduct component
  • have a data property called products that is set to the array of data you’ll find in the products.txt file in the zipfood-assets download.

Here’s a template you can use:

<!-- src/components/ShowProducts.vue -->
<template>
    <div id="products">
        <h2>Products</h2>
        <show-product
            v-for="product in products"
            :key="product.id"
            :product="product"
        ></show-product>
    </div>
</template>

ShowFeatured component

Next, we want to display a list of products within the predefined category of snacks.

Steps:

  • Create a ShowFeatured component that will display a h2 that says Featured Products, followed by a list of products that are categorized as snacks.
  • Define a products data property, just like you did in ShowProducts (If you think this feels redundant, you are correct! We’ll optimize it in the next step)
  • Set up a component prop category that will be used to decide which category of products should be featured.
  • Define a computed property featuredProducts that filters the products based on the category prop: 
// Computed property in src/components/ShowFeatured.vue
featuredProducts() {
    return this.products.filter((product) => {
        return product.categories.includes(this.category);
    }, this.category);
},

Here’s a template you can use for this component:

<!-- src/components/ShowFeatured.vue -->
<template>
    <div id="featured">
        <h2>Featured Products</h2>
        <ul class="cleanList">
            <li v-for="product in featuredProducts" :key="product.id">
                {{ product.name }}
            </li>
        </ul>
    </div>
</template>

And here’s an example of how this component could be invoked:

<!-- src/App.vue -->
<show-featured :category='"snacks"'></show-featured>

Address redundant data with props

Right now the products data is hard-coded in both the ShowProduct and ShowFeatured components.

Refactor this so that the data is defined in the root App.vue component and passed as a prop to each component.

The application should look the same after you’ve completed this step.

Address redundant data with a “single source of truth”

The above props approach worked, but because so many components in this application will rely on our products data, we should think of a better way of sharing this data.

First, let’s extract the data into its own file, /src/products.js, where we’ll define it as a constant:

// src/products.js
export const products = [ ...products go here ... ]

Typically our data comes from some external/compartmentalized source, such as a database or a data API, so this feels like a good step in that direction.

Next, any component that needs access to the data can import it:

import { products } from '@/products.js';

And define it as one of its data properties:

data() {
        return {
            products: products
        };
    },

We won’t be programming any of our components to alter our products data, but if we did, this setup would be ideal because any changes to our products would be reactive across all components because the data was coming from one ”single source of truth”.

FYI

As applications grow in complexity, it’s worth checking out Vuex, a state-management library for Vue. Vuex is overkill for where we’re at right now, though, but we’ll take a look at it in a future lecture.

Recap

This was our warmup. We set up a simple interface, a few components, and introduced some data. We also introduced assets such as images and CSS into our application. Along the way, we reviewed some of the Vue.js concepts we’ve been studying this semester.
