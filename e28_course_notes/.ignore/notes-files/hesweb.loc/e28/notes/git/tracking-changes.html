Tracking changes

With your local repository set up and connected to your remote repository at Github.com, you’re ready to start tracking changes.

To begin, run git status to see the status of your current working directory, for the current branch you’re on. (We won’t utilize git branches in this course, so we’ll always just be on the default branch called master.)

Assuming you haven’t yet made any changes (added/removed/edited files), the command git status should report back something like this: 

On branch master
Your branch is up-to-date with 'origin/master'.
nothing to commit, working directory clean

Now let’s proceed forward and start to make some code/file changes in this repository so we can see how git can track those changes...

Set up your first application

Within our e28 directory, we want to create subdirectories for every unique application we build in this course.

As a starting example, let’s create an application called, simply, Practice. We’ll start this application by creating a new empty subdirectory within our e28 directory.

Example:

/path/to/htdocs/e28 $ mkdir practice

Then, within this practice directory, create a README.md file that will contain information about the application.

Example:

/path/to/htdocs/e28 $ cd practice
/path/to/htdocs/e28 $ nano README.md

Fill this file with the following content:

# Practice Application

Next, let’s create a index file for this application called index.html .

/path/to/htdocs/e28 $ nano index.html

Fill this file with the following content:

<!doctype html>
<html lang='en'>
<head><title>Practice Application</title>
    <meta charset='utf-8'>
</head>
<body>
    <h1>Practice Application</h1>
</body>
</html>

Next, let’s see how we can tell Git to track these changes...

Tracking changes

Within your e28 directory, run git status again and note that git recognizes there are new files in the repository that it is not yet tracking.

/path/to/htdocs/e28 $ git status
On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

	practice/

nothing added to commit but untracked files present (use "git add" to track)

As we can see, git does not automatically track new files/directories in your repository- you have to instruct git to start tracking these files. This is done using the add command.

/path/to/htdocs/e28 $ git add practice

Now, run git status again and you should see you have two new files (from within that practice directory) that are now staged and ready to be committed. 

/path/to/htdocs/e28 $ git status
On branch master

Initial commit

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)

	new file:   practice/README.md
	new file:   practice/index.html

Committing

In order to synchronize these file changes to our main repository at Github.com, we need to create a commit, which will essentially bundle up all our staged changes into a package that can then be delivered/synched with Github.com.

To do this, we'll use the commit command plus the -m flag which indicates we’ll also be including a message that summarizes what is happening in the commit:

/path/to/htdocs/e28 $ git commit -m "Added practice application"

Finally, we need to push this commit to Github.com using the git push command:

/path/to/htdocs/e28 $ git push

Double check your commit made it to Github.com by refreshing your repository's page.

Practice

Let’s run through that procedure again, after making the following changes:

  • Make an edit to the README.md file in the root of your repository
  • Make an edit to the README.md file within the practice subdirectory
  • Make an edit to the index.html file within the practice subdirectory
  • Create a new file contact.html file within the practice subdirectory
  • Put a image file within a new directory at practice/images

After making these three changes, run git status to see the results.

FYI

You can use the command git diff on individual files if you need a reminder of changes were made to a file. E.g.:

$ git diff README.md

Assuming you’re happy with all your changes, stage them using the add command again. You can stage the files one at a time:

/path/to/htdocs/e28 $ git add README.md
/path/to/htdocs/e28 $ git add practice/README.md
/path/to/htdocs/e28 $ git add images
/path/to/htdocs/e28 $ git add index.html
/path/to/htdocs/e28 $ git add contact.html

Or you can append the --all flag to the git add command to stage all changed/new files:

/path/to/htdocs/e28 $ git add --all
FYI

Note that the git add command is used to not only stage new files, but also stage changes/edits to existing files.

Once your changes are staged, do another commit:

/path/to/htdocs/e28 $ git commit -m "Updated root README; Misc. changes to Practice project"

Finally, push your commit:

/path/to/htdocs/e28 $ git push

Check your repository in Github.com to make sure the commit is there.

Summary

  1. add a.k.a stage any changed/new files
  2. Create a commit of files that have been staged
  3. push the commit to Github.com

Caution: Editing files locally vs. directly on Github.com

It’s possible to edit files directly on Github.com via their web-based editing interface. If you do this, you need to run git pull in your repository locally so that the code base is brought into sync with changes made on Github. It’s especially important you do this before making any additional changes locally, otherwise you may run into merge conflicts.

That being said, I generally recommend you only edit files locally to avoid confusion and merge conflicts. Think of the path between your local code and Github.com as a one way street— It’s possible to go backwards down that street, but doing so can cause problems.

Command recap

add

In Git, you have to add file contents to your staging area before you can commit them. This command is used to track new files, as well as edits to already-tracked files.

commit

A commit is used to bundle together changes that have been staged in a repository. This may include tracking of new files, edits of already-tracked files, or removal of previously tracked files. Commits should be made with a commit message that includes a brief description of what changes were made.

push

Pushing refers to sending your committed changes to a remote repository such as GitHub.com.

pull

Pull is the opposite of push, and is used to retrieve commits from a remote repository, bringing your workin repository into sync with the remote repository. We’ll use this command once we set up our production servers and need to pull code changes there.

Tips

  • If you get a message about src refspec master does not match any. after trying to push— it means you have not yet created a commit.

  • If you want to undo changes made to a file that haven’t yet been staged, you can use the command git checkout <file>.

  • If you stage a file but decide you want to unstage it, run this command: git reset HEAD <file>. This won’t revert any changes to that file (that’s what checkout does), it will just remove it from staging.

  • If you delete a file from a project, you’ll use the git rm <file> command instead of git add <file>.

  • git status is a command you should take advantage of frequently; it will give you a sense of what’s going on in your repository: what has changed, if any files are untracked, etc.

  • git diff is another useful command to show what changes have been made. You can see all changes with just git diff or see just the changes on a particular file with git diff <file>.
