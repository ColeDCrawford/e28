e28 Course Introduction

Single-page Applications & Interfaces w/ Vue.js

Website architecture

Multi-page applications (MPA)

The traditional (and perhaps most straightforward) form of a dynamic, data-driven site can be constructed as a multi-page application:

  • Start with a web page with a form (e.g. search box on Wikipedia)
  • The form is submitted to a separate page that processes the form and displays the results

We describe this as a multi-page application because the processing requires we reload the browser page for each action.

  • Pros: Straightforward
  • Cons: Requires a full page refresh to process the form and display the results

This is acceptable for “heavier” actions:

  • Submitting an order at an e-commerce store
  • Updating your profile information on a social network

But is cumbersome for “lighter” actions such as:

  • Adding an item to your cart at an e-commerce store
  • Submitting a comment on or liking a friend’s post on a social network

Single-page applications (SPA)

Enter single-page applications, where instead of relying on a page-reload, we handle all the processing within a single page via the powers of JavaScript.

With JavaScript, we can send a request to the server with the details of our form (or whatever action the user is taking), receive a response from the server, and display the results on the page— all within the same single page, with no refreshes.

This creates a quicker and more responsive user experience.

Example: Gmail

Note how SPA sites often imitate MPA expectations, e.g. updating the URL bar as you navigate about.

Hybrid

Many sites are hybrids - using a combination of MPA and SPA techniques.

Example: Trello

Regarding this, when I refer to SPA I don’t always exclusively mean an absolutely single page web site. Oftentimes, I mean it more generally speaking in regards to SPA-techniques.

Frameworks

At the very least, a SPA can be built with HTML, CSS, and JavaScript.

But SPAs can be complex, and front-end frameworks exist to make building and maintaining SPAs easier.

  • Angular (by Google)
  • React (by Facebook)
  • Vue (by Evan You who worked with AngularJS at Google)

The role of the server

SPAs offload a lot of the work the server traditionally did in MPA’s to the browser/JavaScript.

The server is still necessary, its role is just different. In a SPA setup, the server is used as an API, or Application Programming Interface to do the things that JavaScript/the browser can’t do such as:

  • Interacting with a database
  • Sending emails
  • Processing credit card payments
  • Etc.

In basic terms, an API allows applications to communicate with one another. So when your application’s JavaScript needs to, for example, store some data in the database, it does so by sending a request to the server which then communicates with the database.

FYI

If you want to learn more about programming the server-side of things, check out my course CSCI E-15 Web Server Frameworks with Laravel/PHP.

Static sites

To conclude this overview of web site architectures, it’s worth mentioning static site generators which have become popular recently.

Static site generators pre-compile web pages, producing a static page of HTML/CSS code that can load very quickly with little to no server-side processing.

Traditional example:

  • A blog web site receives a request to show a particular post.
  • The server processes the request, querying the database for the necessary info (post content, title, author, tags, etc.), constructing the necessary HTML and CSS response which is returned to the browser.
  • This process repeats for every request to that post.

Static site example:

  • When the blog owner creates or updates a post, they run a process that pre-compiles a page of HTML and CSS code for that post. The static result is stored on the server.
  • The blog site receives a request to show that post, and the static page that was pre-compiled is returned. No extra processing or database querying required.
  • SPA features can be incorporated into the post page via “microservices” to provide additional functionality (e.g. comments).
