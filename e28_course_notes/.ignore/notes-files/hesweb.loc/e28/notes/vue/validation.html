Validation

When first studying communicating with the server, we saw how the APIs we’re using may report back some validation errors that can be communicated in our interface.

For example, if a user tried to add a new product without specifying a product name, the server API request would return an error indicating the the name field was required.

We can describe this as sever-side validation since the validation is happening on the server.

This sort of server-side validation is important, but it’s often advisable to use it in conjunction with some pre-submission client-side validation.

In summary, your full validation process should look something like this:

  • Client-side validation is first used to give the visitor instant feedback on their data, and to avoid making an unnecessary trip to the server with bad data that will only get rejected.
  • Assuming client-side validation passes, data is then posted to the server where it does server-side validation to confirm the data is as expected. This ensures that there’s no “funny business” by the visitor in terms of overriding your JavaScript validation in the client.

With that in mind, let’s explore how we can implement client-side validation.

Client-side validation techniques

A quick and easy way to validate form inputs client-side is to add HTML5 validation attributes to your form inputs.

For example, you can add a required attribute to a text input:

<input type='text' id='sku' v-model='product.sku' required>

This will prevent a form submission if the given field is empty, and it will alert the user that the input needs the be completed:

(For more on HTML5 validation attributes, read the section *Using built-in form validation here: MDN: Form data validation)

HTML5 validation works to a certain extent, but you may want more control with the ability to have custom validation messages, behaviors, and rules.

Because of this, you should consider some more robust validation approaches including...

  1. Write your own validation functionality from scratch (example: Vue.js cookbook: Form validation)
  2. Write your own validation functionality with the assistance of a generic JavaScript validation library such as validatorjs).
  3. Use a Vue-specific validation library with many “bells and whistles” such as Vuelidate.

For our example, we’ll take the middle ground and explore option #2.

validatorjs

The package validatorjs is a JavaScript validation library that offers some basic validation logic and a library of validation rules.

To get started, install it in your project:

$ npm install --save validatorjs

And import it in the component you want to add validation to:

import Validator from 'validatorjs'

For our example, we’ll add client-side validation to our ProductCreatePage component, so that’s where you’ll want to import it.

Next, add a validate method with the following code:

validate() {
    let validator = new Validator(this.product, {
        name: 'required|between:3,100',
        sku: 'required|between:3,100|alpha_dash',
        price: 'required|numeric',
        available: 'required|numeric',
        weight: 'required|numeric',
        description: 'required|min:100'
    });

    if (validator.fails()) {
        this.errors = validator.errors.all();
    } else {
        this.errors = null;
    }

    return validator.passes();
},

Observations:

  • The Validator is instantiated by passing it an object to validate (in our case, this.product) and an object of rules to validate against.
  • In this example we see rules such as required, between, numeric, alpha_dash, and min. See validatorjs#Available Rules for a full list of available rules.
  • Multiple rules are separated by a pipe character (|)
  • Some rules require values, e.g. between:3,100 and min:100
  • We retrieve any errors from the validator via validator.errors.all(). These errors are stored in our existing errors data property, which means our interface can now display client-side validation errors and server-side validation errors.

To use our new validate method add a blur event listener to each of the fields under validation. E.g.:

<input
    type="text"
    v-model="product.name"
    id="name"
    @blur="validate()"
/>

You can also update the existing addProduct method so it will only attempt to add a new product on the server if client-side validation passes:

addProduct() {
    if (this.validate()) {
        // code here to ping the server API with a "add a product" request
    }
}

Test it

  • When leaving each of the fields, validation should be triggered and you should see immediate feedback of issues.
  • If there are any errors, clicking the Add Product button should not do anything (you could even write code to make the button inactive if there are errors)
